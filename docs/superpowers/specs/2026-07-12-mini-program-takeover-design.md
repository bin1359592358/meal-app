# 「超大一碗饭」小程序接管与重构设计

## 1. 项目目标

本次改造将「超大一碗饭」调整为只维护微信小程序和 FastAPI 后端的项目，停止 Android 客户端开发。改造同时解决现有逻辑问题、补齐缺失功能，并建立以现有厨娘头像为核心的轻二次元品牌视觉。

验收时，用户应能稳定完成以下闭环：

1. 微信静默登录或用户名 + PIN 登录；
2. 创建或加入餐桌；
3. 浏览、搜索、随机选择菜品；
4. 查看菜品详情并选择调味；
5. 管理购物车并提交订单；
6. 查看订单和取消待处理订单；
7. 主厨管理分类、菜品、成员和订单状态；
8. 主厨查看口径明确的订单与营业统计；
9. 在多个餐桌之间切换，不残留旧菜单、旧角色或旧购物车数据。

## 2. 项目边界

### 2.1 保留内容

- 微信原生小程序：WXML、WXSS、JavaScript；
- FastAPI、SQLAlchemy 和 Pydantic 后端；
- SQLite 本地开发和 Turso 云端数据库；
- Railway、Cloudflare Worker 和现有自定义域名；
- Cloudinary 或本地图片上传；
- 现有数据库表和 API 主路径。

### 2.2 移除内容

- 完整删除 `android/` 目录；
- 删除 README 和架构文档中的 Android 构建、配置及功能说明；
- 不再维护 Kotlin、Jetpack Compose、Gradle、APK 和 Android 签名配置；
- 不将 Android 遗留设计作为小程序重构约束。

### 2.3 不引入内容

- 不引入 Node.js 构建链；
- 不引入第三方小程序 UI 框架；
- 不引入复杂全局状态库；
- 不在本轮更换后端框架或数据库模型体系；
- 不进行与点菜闭环无关的功能扩张。

## 3. 总体架构

正式运行端只保留以下数据流：

```text
微信小程序
    ↓ HTTPS
Cloudflare Worker
    ↓
Railway FastAPI
    ↓
Turso / SQLite
```

小程序继续使用原生页面和组件，但将重复业务逻辑提取到共享模块：

- `app.js`：登录态入口、全局购物车和应用级事件；
- `utils/session.js`：登录失效、当前餐桌变化和本地状态清理；
- `utils/cart.js`：餐桌隔离、购物车项目唯一键和数量规则；
- `utils/seasoning.js`：调味默认值、校验、规范化和摘要格式化；
- `utils/api.js`：统一请求、上传、错误解析和 401 处理；
- 页面：只负责页面数据、生命周期和用户交互。

后端继续作为权限、价格、订单状态和调味合法性的最终可信来源。

## 4. 功能设计

### 4.1 登录与会话

- 保留微信静默登录和用户名 + PIN 备选登录；
- 401 响应统一清除用户、餐桌和购物车，再返回登录页；
- 退出登录先尽力通知服务端，再清除本地状态；
- 空 `pin_hash` 用户不能触发 bcrypt 异常；
- PIN 修改成功后，客户端保存后端返回的新 Token；
- 生产环境默认禁止自动创建固定密码演示账号，也不输出有效 Token。

### 4.2 餐桌

- 邀请码输入统一去除首尾空格并转换为大写；
- 切换餐桌前，如购物车非空，提示用户旧购物车将被清空；
- 切换后立即清空购物车并刷新菜单、订单和角色；
- 被移除、主动退出或餐桌关闭后，删除失效缓存；
- 如仍有可用餐桌，自动选择一个；否则进入创建或加入流程；
- 关闭操作保持幂等，不允许通过重复点击重新打开；
- 主厨移除成员时使用 `user_id`，而不是成员关系记录 ID。

### 4.3 菜单、详情与搜索

- 菜单首页使用横向分类胶囊和纵向菜品列表；
- 点击菜品卡片打开详情与调味合并面板；
- 面板展示图片、描述、标签、价格、库存状态和调味项；
- 搜索使用防抖和请求序号，旧请求不得覆盖新结果；
- 随机选菜仅从当前餐桌的可用、未删除菜品中选择；
- 售罄菜品不能通过卡片、详情或随机推荐加入购物车；
- 菜品列表、搜索和随机推荐共享同一份可用性规则。

### 4.4 调味和购物车

- 支持单选、多选、滑块和文本输入；
- 同一菜品的不同调味组合必须成为独立购物车项目；
- 唯一键由菜品 ID 和规范化调味 JSON 组成；
- 购物车只属于一个餐桌；
- 切换餐桌、退出餐桌或退出登录时清空购物车；
- 调味摘要在菜单、购物车、订单详情和主厨总览中使用统一格式；
- 数量范围与后端保持一致，单项最多 999 份；
- 下单价格由后端重新计算，客户端价格仅用于预览。

### 4.5 订单

- 修复小程序读取 `seasoning_text` 的字段不一致问题；
- 食客只能取消自己的待处理订单；
- 主厨按 `pending → preparing → served → completed` 推进状态；
- `pending` 和 `preparing` 可以取消，其他非法转换由后端拒绝；
- 订单分页需要防止并发加载、重复页和重复订单；
- 当前餐桌的待处理订单数量用于 TabBar 角标；
- 时间、金额、人均价和状态文案使用统一格式化工具。

### 4.6 主厨管理

- 分类删除提示与后端规则一致：分类中存在未删除菜品时禁止删除；
- 菜品编辑正式支持 `is_available`；
- 菜品编辑补齐调味配置器；
- 支持菜品名称、描述、价格、分类、标签、图片、排序和上下架；
- 图片上传失败时保留表单内容；
- 表单提交和状态切换需要防重复点击；
- 当前用户不是主厨时，页面入口隐藏，后端权限仍作为最终防线。

### 4.7 统计

统计界面拆分为两个口径：

- **进行中：** `pending`、`preparing` 和 `served`；
- **经营统计：** 已完成订单和有效营业金额，不包含 `cancelled`。

菜品聚合使用菜品 ID 和菜品名称，避免同名菜品合并。历史订单继续使用下单时保存的菜名、单价和调味快照。

## 5. 后端可靠性和校验

- 根据数据库 URL scheme 判断 SQLite、libSQL 或其他数据库，不能只检查 Turso 环境变量；
- 菜品、分类、餐桌、昵称、备注、标签和调味文本增加明确长度限制；
- 上传图片同时校验 MIME、扩展名、大小和文件头；
- 调味选择必须与菜品定义匹配，拒绝未知调味项、未知选项和超长文本；
- 菜品更新 schema 支持 `is_available`；
- 创建房间邀请码继续使用 CSPRNG，并处理唯一键冲突；
- 关键写操作捕获数据库异常、回滚事务并返回明确错误；
- 保留订单金额的服务端重算和关闭餐桌后的下单拦截。

## 6. 视觉设计

### 6.1 视觉定位

最终定位是「以现有头像角色为核心的轻二次元厨娘主题 UI」，不是大头小身体的 Q 版风格。

禁止使用：

- 二头身、三头身或 super-deformed 角色；
- 幼儿绘本式人物；
- 大量 emoji 和贴纸堆叠；
- 幼圆字体和玩具式组件；
- 将厨娘角色简化成大头吉祥物。

### 6.2 角色规范

保留以下身份特征：

- 银白色双丸子长发；
- 蓝色眼睛；
- 白色厨师帽；
- 蓝白厨师服和蓝色蝴蝶结；
- 厨具和厨娘主题；
- 奶黄背景、柔和光线和日系插画质感。

根据场景使用头像特写、胸像、半身像或正常比例全身立绘。所有生成提示必须包含「正常日系动漫人物比例，禁止 chibi，禁止 super-deformed，禁止大头小身体」。

### 6.3 配色和组件

- 主色：厨娘蓝 `#5579BC`；
- 深色操作：深蓝 `#35588F`；
- 选中背景：雾蓝 `#EAF1FF`；
- 页面背景：奶油白 `#FFFDF8`；
- 暖色点缀：奶黄 `#FFDFA0`；
- 情绪强调：腮红粉 `#F6B9B5`；
- 正文：蓝灰黑 `#30394A`；
- 次级文字：灰蓝 `#7B8496`。

组件保持现代、高效和克制。二次元氛围主要由角色插画、柔光、局部蓝白格纹、蝴蝶结和厨师帽轮廓承担。

### 6.4 页面结构

- **菜单：** 品牌与餐桌头部、搜索、横向分类、菜品列表、底部购物车；
- **菜品详情：** 图片、描述、标签和调味合并为底部面板；
- **订单：** 状态筛选、订单卡片和清晰的状态操作；
- **我的：** 用户信息、餐桌切换、邀请和成员；
- **主厨工作台：** 菜单管理、订单统计和餐桌设置集中入口。

## 7. 图片资源策略

首轮正式资源包括：

- 头像压缩版；
- 登录页品牌图；
- 小程序分享封面；
- 3～4 张通用状态插画；
- 风格统一的 TabBar 图标。

图片不承载需要动态变化的中文文字。所有文字由 WXML 渲染。生成资源需要压缩并检查尺寸、透明通道、可读性和包体影响。

## 8. 状态和错误体验

- 加载中使用轻量厨师帽、饭碗或柔和骨架屏；
- 空状态使用角色插画、原因说明和单一主操作；
- 售罄、餐桌关闭等业务错误优先页面内提示；
- 网络失败保留用户尚未提交的表单；
- 登录失效只执行一次全局跳转，避免多个页面重复弹窗；
- 下单、状态更新和上传期间锁定对应操作，防止重复提交。

## 9. 测试策略

### 9.1 后端测试

使用 `pytest` 和 FastAPI `TestClient`。测试直接调用真实路由和数据库逻辑，但使用临时 SQLite 数据库：

- 不需要预先运行 `python main.py`；
- 不访问 Railway 或正式 Turso；
- 不污染 `backend/meal_app.db`；
- 测试结束自动清理临时数据。

覆盖登录、权限、餐桌、菜品、调味、购物车请求、订单状态、统计和上传校验。

### 9.2 小程序测试

将购物车、调味、格式化和会话清理提取为纯 JavaScript 模块，使用 Node.js 内置测试工具验证。页面层通过语法检查、事件绑定检查和微信开发者工具人工流程验证。

### 9.3 验证命令

最终至少运行：

```powershell
python -m pytest backend/tests -v
node --test miniprogram/tests/*.test.js
Get-ChildItem miniprogram -Recurse -Filter *.js | ForEach-Object { node --check $_.FullName }
python -m compileall -q backend
```

## 10. 实施顺序

1. 建立隔离测试基线；
2. 删除 Android 并清理文档；
3. 修复后端配置、安全、校验和统计；
4. 重构小程序会话、购物车和调味工具；
5. 修复全部已知业务问题；
6. 补齐随机选菜、详情面板和调味编辑；
7. 制作并接入品牌资源；
8. 按菜单、购物车、订单、个人中心、主厨工作台顺序重做 UI；
9. 完整回归核心链路；
10. 更新 README、架构和部署说明。

## 11. 完成标准

- 工作区不再包含 Android 源码和构建说明；
- 已知逻辑问题均有回归测试或明确验证步骤；
- 多餐桌切换不会残留菜单、角色或购物车；
- 调味组合在购物车和订单中保持一致；
- 主厨成员、菜品和订单操作符合权限规则；
- 统计口径与页面文案一致；
- 从登录到订单完成的核心链路可以完整运行；
- 页面视觉统一为正常比例二次元厨娘风格；
- 生产环境不自动暴露固定密码演示账号或有效 Token。
