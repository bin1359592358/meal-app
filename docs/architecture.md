# 超大一碗饭架构说明

## 1. 系统范围

「超大一碗饭」采用前后端分离架构，仅包含以下运行组件：

1. **微信小程序：** 面向主厨与食客，负责页面展示、交互、本地会话和 API 调用。
2. **FastAPI 后端：** 负责认证、权限校验、业务规则、数据持久化和图片上传。

历史设计规格与实施计划保存在 `docs/superpowers/specs/` 和 `docs/superpowers/plans/`，用于记录决策背景，不作为当前运行架构说明。

## 2. 总体架构

```text
┌──────────────────────────────┐
│          微信小程序           │
│ 页面 / 组件 / 本地存储 / 请求 │
└──────────────┬───────────────┘
               │ HTTPS + JSON
               │ Bearer Token
┌──────────────▼───────────────┐
│        FastAPI 后端           │
│ 路由 / 认证 / 权限 / 业务规则 │
└───────┬──────────────┬───────┘
        │              │
┌───────▼───────┐ ┌────▼────────────┐
│ SQLite / Turso│ │ Cloudinary（可选）│
│ 业务数据       │ │ 菜品图片           │
└───────────────┘ └─────────────────┘
```

### 核心领域

- **用户：** 保存用户名、昵称和认证信息。
- **餐桌：** 由主厨创建，食客通过邀请码加入。
- **分类与菜品：** 主厨维护菜单、价格、标签、调味项和可售状态。
- **订单：** 保存下单人、菜品快照、数量、调味、人数、备注和状态。
- **成员关系：** 连接用户与餐桌，并记录 `chef` 或 `guest` 角色。

## 3. 微信小程序

### 目录职责

```text
miniprogram/
├── pages/          # 页面及页面状态
├── components/     # 菜品卡片、购物车栏、订单卡片等通用组件
├── utils/
│   ├── api.js      # 请求封装与图片上传
│   ├── auth.js     # 登录态恢复与退出
│   ├── storage.js  # Token、用户、餐桌和服务地址存储
│   └── util.js     # 通用工具
├── app.js          # 应用启动逻辑
├── app.json        # 页面、导航和组件配置
└── app.wxss        # 全局样式
```

### 页面边界

| 页面 | 当前代码职责 |
| --- | --- |
| `login` | 账号登录、注册、创建餐桌和加入餐桌 |
| `menu` | 分类与菜品浏览、搜索、加购入口 |
| `cart` | 数量调整、人数与备注、提交订单 |
| `orders`、`order-detail` | 订单列表、详情与状态操作 |
| `profile` | 用户资料、餐桌成员与退出操作 |
| `admin`、`dish-edit` | 分类和菜品维护 |
| `overview` | 按菜品聚合订单 |

这些页面代表现有代码边界。界面完整性、真机兼容性和全流程可用性仍以微信开发者工具及真机验收结果为准。

### 请求与会话

1. `utils/storage.js` 保存服务地址、Token、用户和当前餐桌。
2. `utils/api.js` 组合服务地址与 `/api` 路径，并在请求头携带 Token。
3. 后端返回未认证响应时，前端清理失效会话并引导重新登录。
4. 小程序启动时由 `utils/auth.js` 尝试恢复登录态。

Token 属于敏感信息，只应保存在小程序本地存储中，不应写入仓库或日志。

## 4. FastAPI 后端

### 目录职责

```text
backend/
├── main.py          # 应用创建、CORS、路由注册、健康检查
├── config.py        # 环境变量
├── database.py      # 数据库引擎与会话
├── models.py        # SQLAlchemy 模型
├── schemas.py       # Pydantic 输入输出模型
├── auth.py          # PIN 哈希与 Token 工具
├── middleware.py    # 认证依赖
├── image_upload.py  # Cloudinary 适配
├── routers/         # 领域路由
├── tests/           # pytest 自动化测试
└── seed_data.py     # 本地演示数据
```

### API 分组

所有业务接口统一使用 `/api` 前缀。

| 分组 | 主要职责 |
| --- | --- |
| `/api/auth` | 注册、账号登录、微信登录、修改 PIN、退出 |
| `/api/users` | 当前用户查询与更新 |
| `/api/rooms` | 餐桌创建、加入、成员、邀请码与关闭状态 |
| `/api/rooms/{room_id}/categories` | 分类查询与维护 |
| `/api/rooms/{room_id}/dishes` | 菜品查询、搜索、维护与上下架 |
| `/api/rooms/{room_id}/orders` | 下单、列表、详情、汇总与状态更新 |
| `/api/upload` | 图片上传 |
| `/api/health` | 部署健康检查 |

后端对需要登录的接口验证 Bearer Token，并在餐桌写操作中校验成员身份与角色。权限规则应以后端校验为准，前端隐藏按钮只用于改善交互，不能代替服务端鉴权。

## 5. 数据模型与一致性

| 模型 | 说明 |
| --- | --- |
| `User` | 用户身份与资料 |
| `Session` | 登录 Token 及有效期 |
| `Room` | 餐桌、邀请码和创建者 |
| `RoomMember` | 用户与餐桌的角色关系 |
| `Category` | 餐桌内的菜单分类 |
| `Dish` | 菜品、价格、标签、调味和可售状态 |
| `Order` | 订单头、人数、备注、总价和状态 |
| `OrderItem` | 菜品快照、数量、调味和小计 |

订单明细保存下单时的菜名、价格和调味快照，避免后续修改菜单影响历史订单。菜品删除采用业务层保护策略，历史订单不依赖实时菜品内容展示。

## 6. 配置与部署

### 数据库

配置优先级为：

1. `DATABASE_URL`
2. `TURSO_DATABASE_URL` 与 `TURSO_AUTH_TOKEN`
3. 本地 `sqlite:///./meal_app.db`

本地数据库及其备份不进入版本库。生产环境不应依赖部署平台的临时文件系统。

### 外部服务

- **微信登录：** 需要 `WX_APPID` 与 `WX_SECRET`。
- **图片存储：** 需要完整的 Cloudinary 配置；未配置时不能把本地上传目录视为生产持久化方案。
- **服务端密钥：** 生产环境必须设置随机且保密的 `SECRET_KEY`。

### Railway

`railway.toml` 使用 Gunicorn 与 Uvicorn Worker 启动后端，并通过 `/api/health` 执行健康检查。发布后，小程序应使用 HTTPS 服务地址，并在微信公众平台配置合法请求域名。

## 7. 开发与验证边界

提交影响运行代码的变更前，至少执行：

```powershell
python -m compileall -q backend
python -m pytest backend/tests
Get-ChildItem miniprogram -Recurse -Filter *.js |
  ForEach-Object { node --check $_.FullName }
```

自动化测试与语法检查不能替代以下验收：

- 微信开发者工具编译与网络请求检查。
- 主厨和食客两个角色的权限边界。
- 创建或加入餐桌、选菜、下单、查看订单的完整流程。
- 微信登录、图片上传和生产数据库的真实外部服务联调。

## 8. 演进原则

- 运行架构保持为微信小程序与 FastAPI 后端两个组件。
- 新功能优先沿现有页面、组件和领域路由边界扩展。
- 权限、金额与订单状态转换必须由后端最终裁决。
- 外部服务均通过环境变量配置，不在源码中保存密钥。
- 历史规格与实施计划保留用于追溯；正式文档以本文件和 README 为准。
